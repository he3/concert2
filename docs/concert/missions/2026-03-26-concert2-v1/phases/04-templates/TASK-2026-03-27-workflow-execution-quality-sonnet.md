## Objective

These are the most operationally critical workflows — they define exactly how tasks are executed, how the orchestrator-coder-reviewer loop works, and the rules for iteration, exit conditions, and state management. ## Detailed Instructions

### CONCERT-WORKFLOW-EXECUTION.md

This workflow governs the entire execution phase — from starting phase 1 through completing all phases.

**Content sections:**

1. **Phase execution order:**
   - Phases execute in numerical order (01, 02, 03, ...)
   - Within each phase, tasks execute in wave order (wave 1 before wave 2)
   - Within a wave, tasks with no mutual dependencies can run in parallel (in GitHub Actions) or sequentially (in Claude Code)

2. **Task execution loop (per task):**
   - Read task file frontmatter for model tier, dependencies, wave
   - Verify all depends_on tasks are complete
   - Execute through the code quality loop (CONCERT-WORKFLOW-CODE-QUALITY.md)
   - On success: update state.json (tasks_completed++, telemetry record, history entry)
   - On failure: write failure block to state.json, stop immediately
   - Update PHASE-SUMMARY after each task file completes

3. **State management rules:**
   - state.json is updated and committed after every task completion
   - current_phase, current_task_file, current_task_index track position
   - quality_loop_state tracks mid-loop position for crash recovery
   - quality_loop_state is cleared after task completion

4. **Phase completion:**
   - After all tasks in a phase complete: spawn concert-documenter
   - Documentation agent updates higher-level docs
   - Mark phase complete in state.json (phases_completed++)
   - Update WIP PR body

5. **Failure handling:**
   - On failure: write failure block with error_type, error_summary, files_touched, last_successful_commit
   - Append to failure_log
   - Stop execution immediately — no skipping ahead
   - Recovery: `concert-debug` or `concert-continue` (which retries)

6. **Context compaction rules:**
   - Current phase: read TASK files + PHASE-SUMMARY
   - Previous phase: read PHASE-SUMMARY only
   - Older phases: skip unless specifically relevant
   - Orchestrator keeps context to ~15% of window

7. **Commit format:**
   - Each task produces exactly one conventional commit
   - Format: `type(scope): description`
   - State.json update is included in the task commit

8. **Cross-environment handoff:**
   - If session stops: state.json reflects last committed task
   - User runs `npx @he3-org/concert push` to push to origin
   - `concert-continue` in other environment resumes from state.json

### CONCERT-WORKFLOW-CODE-QUALITY.md

This workflow governs the orchestrator-coder-reviewer loop for a single task.

**Content sections:**

1. **Loop structure:**
```
Orchestrator (concert-continue)
  |
  |--> Spawn Coder (model from task frontmatter)
  |      |- Read task file, applicable skills
  |      |- TDD: write tests -> implement -> verify all tests pass
  |      |- Commit with conventional format
  |      |- Return: commit SHA, confidence, files changed
  |
  |--> Spawn Reviewer (balanced model)
  |      |- Read task file, diff from coder's commit
  |      |- Review against acceptance criteria
  |      |- Rate findings: CRIT / MAJ / MIN / NTH / PASS
  |      |- Return: structured findings list
  |
  |--> Decision:
         |- PASS (zero findings): exit loop, task succeeds
         |- Any findings AND iteration < max: re-spawn Coder with findings
         |- After max iterations, only MIN/NTH remaining: exit with success
         |- After max iterations, CRIT/MAJ remaining: exit with FAIL
```

2. **Coder behavior:**
   - Always TDD: tests first, then implementation, then verify
   - On revision (iteration > 1): receive prior findings, address them in priority order
   - CRIT and MAJ must be resolved first
   - If only MIN/NTH: resolve as many as possible
   - Produce exactly one commit per iteration
   - Update inline documentation (JSDoc, comments) within touched files

3. **Reviewer behavior:**
   - Review the diff against the task's acceptance criteria
   - Each finding has: severity (CRIT/MAJ/MIN/NTH), file, line, issue description
   - Severity definitions:
     - CRIT: Security vulnerability, data loss risk, crash bug
     - MAJ: Incorrect behavior, missing requirement, test gap
     - MIN: Code quality issue, naming, missing docs
     - NTH: Style nit, optional improvement
     - PASS: No findings — code meets all acceptance criteria
   - Reviewer uses the balanced model regardless of the coder's model tier

4. **State tracking (quality_loop_state):**
   - Written to state.json at the start of each loop iteration
   - Fields: task_file, task_index, iteration, phase (coder/reviewer), prior_findings, coder_commits
   - Cleared when the task completes (success or fail)
   - Enables crash recovery: `concert-continue` reads this to resume mid-loop

5. **Exit conditions (detailed):**
   - PASS at any iteration: SUCCESS — proceed to next task
   - After max_review_iterations with only MIN/NTH: SUCCESS — proceed
   - After max_review_iterations with any CRIT/MAJ: FAIL — stop execution
   - On FAIL: output "Run `concert-continue --max-iterations N` to allow more iterations"

6. **Telemetry record (on success):**
   - task_file, task_index, phase, model_assigned, confidence, review_result, revision_count, skills_loaded, files_changed, completed_at
