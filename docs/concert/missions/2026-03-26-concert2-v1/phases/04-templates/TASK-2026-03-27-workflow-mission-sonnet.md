## Objective

Write the three mission workflow definition files that define the orchestration rules for different feature sizes. These workflows are the source of truth for which pipeline stages are active, which agents are assigned to each stage, and the rules for stage transitions. Agents read these files at runtime to determine their behavior.

## Files

- `templates/docs/concert/workflows/CONCERT-WORKFLOW-MISSION-FULL.md`
- `templates/docs/concert/workflows/CONCERT-WORKFLOW-MISSION-MEDIUM.md`
- `templates/docs/concert/workflows/CONCERT-WORKFLOW-MISSION-SMALL.md`

## Requirements

- FR-030: Workflow Selection by Feature Size
- FR-009: Review/Accept/Restart Cycle
- Architecture Section 5: Workflow Files

## Detailed Instructions

### CONCERT-WORKFLOW-MISSION-FULL.md

This is the large feature workflow. All stages are active.

**Content structure:**

1. **Header and description:** One-line description of when this workflow applies (large features).

2. **Pipeline stages (ordered):**

| Stage | Agent | Interactive | Description |
|-------|-------|-------------|-------------|
| vision | concert-init | yes | Interactive interview to create VISION.md |
| requirements | concert-analyst | no | Generate REQUIREMENTS.md from accepted vision |
| architecture | concert-architect | no | Generate ARCHITECTURE.md from accepted requirements |
| ux | concert-designer | no | Generate UX.md from accepted architecture |
| tasks | concert-planner | no | Decompose approved plans into task files |
| execution | concert-continue | no | Execute task files through the code quality loop |
| verification | concert-verifier | no | Verify completed work against requirements |

3. **Stage transition rules:**
   - Each stage produces a draft document
   - Draft must be reviewed (`concert-review`) and accepted (`concert-accept`) before advancing
   - On accept: document is copied to `*-SPEC.md` at project level
   - On restart: draft is discarded, stage re-runs from scratch
   - Pipeline advances linearly — no stage skipping in normal flow
   - `concert-replan <stage>` marks downstream stages as needing re-run

4. **Review cycle rules:**
   - Review is two-phase: user changes first, then reviewer concerns
   - Review uses `concert-reviewer` agent
   - Accept uses `concert-accept` (no agent — direct state transition)

5. **Execution rules reference:**
   - Execution follows CONCERT-WORKFLOW-EXECUTION.md
   - Code quality follows CONCERT-WORKFLOW-CODE-QUALITY.md

### CONCERT-WORKFLOW-MISSION-MEDIUM.md

Medium feature workflow. Skips UX stage.

**Active stages:** vision, requirements, architecture, tasks, execution, verification

Same structure as full but with the `ux` stage removed from the pipeline table. Include a note: "UX stage is skipped. Architecture feeds directly into task planning."

### CONCERT-WORKFLOW-MISSION-SMALL.md

Small feature workflow. Only vision, tasks, and execution.

**Active stages:** vision, tasks, execution, verification

Same structure but with requirements, architecture, and UX removed. Include a note: "Planning stages (requirements, architecture, UX) are skipped. Vision feeds directly into task planning. Use for small, well-understood features."
