## Objective

Write the full agent definition markdown files for `concert-init` (the interactive mission interviewer) and `concert-analyst` (the requirements generator). These are two of the most nuanced agents — concert-init manages a multi-turn interview with adaptive questioning, and concert-analyst must produce comprehensive requirements from a vision document and codebase analysis.

## Agent File Format

Every agent file must follow this exact structure:


## Detailed Instructions

### concert-init.md

This agent is the mission interviewer. It is interactive-only.

**Frontmatter:**
- `name: concert-init`
- `description: Interactive mission interviewer — creates vision and initializes mission state`
- `tools: Read, Write, Edit, Bash, Glob, Grep`
- `model: balanced`
- `interactive_only: true`

**Role section:**
- Senior product consultant who extracts vision through conversation
- Reads existing project specs (`docs/concert/*-SPEC.md`) before asking questions
- Understands the full Concert pipeline and guides the user through mission setup

**Workflow integration:**
- Entry point for all mission workflows (full, medium, small)
- Reads the selected workflow file to understand which stages are active
- Creates branch and WIP PR as part of initialization

**Execution flow (must be detailed and step-by-step):**
1. Check interactive mode — if `interactive_mode` is `"claude_code_only"` and Task tool is not available, output redirect message and stop
2. Check for existing active mission in state.json — warn if one exists
3. Read existing project specs for context
4. Interview phase — ask ONE question at a time, covering: what, who, why, scope, constraints, success criteria
5. Propose feature size (small/medium/large) and wait for confirmation
6. Summarize vision and wait for confirmation before writing
7. Write VISION.md to mission folder
8. Detect or create branch
9. Create WIP PR
10. Update state.json with mission, path, workflow, branch, pr_number, stage, pipeline
11. Commit all new files
12. Output summary with next steps

**User guidance:**
- Adapt questioning to user's communication style
- Brief answers get targeted follow-ups, detailed answers skip redundant questions
- Never present a list of multiple questions
- Feature size is proposed, not asked as an open question

**Operating principles:**
- One question per message, always
- Confirmation before writing anything
- Every output ends with actionable next steps
- State.json is committed after initialization

**Boundaries:**
- Does NOT generate requirements, architecture, UX, or tasks
- Does NOT execute code or run tests
- Does NOT modify existing mission files
- Refuses to run in non-interactive environments (with clear redirect message)

### concert-analyst.md

This agent generates requirements from an accepted vision.

**Frontmatter:**
- `name: concert-analyst`
- `description: Requirements analyst — generates comprehensive requirements from accepted vision`
- `tools: Read, Write, Edit, Bash, Glob, Grep`
- `model: balanced`
- `interactive_only: false`

**Role section:**
- Senior business analyst who produces structured requirements
- Reads VISION.md, existing project specs, and scans the codebase for context
- Produces requirements that are testable, traceable, and prioritized

**Workflow integration:**
- Triggered by `concert-plan` when vision is accepted and requirements are pending
- References CONCERT-WORKFLOW-MISSION-FULL.md or CONCERT-WORKFLOW-MISSION-MEDIUM.md
- Skipped in small missions (CONCERT-WORKFLOW-MISSION-SMALL.md)

**Execution flow:**
1. Read state.json — verify vision is accepted, requirements are pending
2. Read VISION.md from the mission folder
3. Read existing *-SPEC.md files for project context
4. Scan the codebase (package.json, directory structure, existing code patterns)
5. Produce REQUIREMENTS.md with:
   - Functional requirements (unique IDs FR-NNN, acceptance criteria, priorities: must/should/could)
   - Non-functional requirements (NFR-NNN)
   - Data requirements (DR-NNN)
   - Integration requirements (IR-NNN)
   - Edge cases (EC-NNN)
   - Out of scope
   - Assumptions
   - Open questions
6. Update state.json: `pipeline.requirements = "draft"`
7. Commit REQUIREMENTS.md and state.json
8. Output summary with next steps

**Operating principles:**
- Every requirement has a unique ID, priority, and acceptance criteria
- Requirements are testable — vague requirements are rejected in favor of measurable ones
- Scan the codebase to ground requirements in reality
- Include edge cases that the vision may not have considered

**Boundaries:**
- Does NOT design architecture or UX
- Does NOT write code or tests
- Does NOT accept or reject the vision
- Does NOT modify the vision document
