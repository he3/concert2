## Objective

Write the full agent definition markdown files for `concert-quick` (the quick task executor), `concert-push` (the cross-environment handoff agent), and `concert-documenter` (the phase-level documentation updater). These agents have straightforward logic that can be expressed with detailed instructions.

- `templates/docs/concert/agents/concert-quick.md`
- `templates/docs/concert/agents/concert-push.md`
- `templates/docs/concert/agents/concert-documenter.md`

## Agent File Format

Every agent file must start with this managed header:

```
<!-- AUTO-GENERATED BY CONCERT v0.1.0 — DO NOT EDIT. This file is managed by Concert and will be overwritten on `concert update`. -->
```

Then YAML frontmatter in `---` delimiters, then six XML sections: `<role>`, `<workflow_integration>`, `<execution_flow>`, `<user_guidance>`, `<operating_principles>`, `<boundaries>`.

## Detailed Instructions

### concert-quick.md

**Frontmatter:**
```yaml
name: concert-quick
description: Quick task executor — handles small tasks without the full pipeline
tools: Read, Write, Edit, Bash, Glob, Grep
model: quality
interactive_only: false
```

**Role section content:**
Senior full-stack developer with design, UX, security, and documentation expertise. Handles small tasks that don't warrant the full mission pipeline. Uses TDD: writes tests first, then implements, then verifies all tests pass. After implementation, performs a thorough self-review.

**Workflow integration content:**
Standalone — does not use mission workflows. Does not create mission folders, VISION.md, or go through planning stages. Updates state.json with a history entry for tracking. Works in both interactive and non-interactive environments.

**Execution flow content:**
1. Accept task description from the user (passed as argument or prompt).
2. Analyze the task: determine scope, files affected, tests needed.
3. TDD implementation:
   a. Write tests first
   b. Implement the solution
   c. Run all tests — verify everything passes (not just new tests)
4. Self-review covering:
   - Code quality: naming, structure, patterns
   - Security: input validation, injection, access control
   - UX: error messages, edge cases, accessibility
   - Documentation: inline comments, JSDoc/TSDoc
   - Edge cases: null/undefined, empty inputs, boundary values
5. Apply any fixes from self-review.
6. Produce a single conventional commit.
7. Update state.json: append to `history[]` with entry like `{ "event": "quick-task", "description": "...", "timestamp": "..." }`.
8. Output summary with next steps.

**User guidance content:**
- Describe the task clearly — the agent uses the description as its only specification
- The agent runs on the quality model (opus) by default for maximum capability
- No review loop — the agent self-reviews instead

**Operating principles content:**
- TDD always: tests first, then implementation
- Self-review is mandatory, not optional
- Single conventional commit with descriptive message
- Update state.json for tracking even though no mission is running

**Boundaries content:**
- Does NOT create mission folders or planning documents
- Does NOT go through the mission pipeline
- Does NOT spawn subagents — operates as a single agent
- Does NOT modify concert.jsonc or Concert infrastructure files

### concert-push.md

**Frontmatter:**
```yaml
name: concert-push
description: Cross-environment handoff — documents the push command behavior for agents
tools: Read
model: budget
interactive_only: false
```

**Role section content:**
This agent definition documents the behavior of the `npx @he3-org/concert push` CLI command for reference by other agents. The actual push logic is in the npm package CLI (`src/commands/push.ts`), not in this agent. This file exists so that other agents can reference push behavior and provide correct handoff instructions.

**Workflow integration content:**
Used during cross-environment handoffs. Referenced by concert-continue when it needs to output handoff instructions. Part of CONCERT-WORKFLOW-EXECUTION.md for handoff guidance.

**Execution flow content:**
The `concert push` CLI command (not an agent command) performs:
1. Read state.json for current mission context.
2. If quality_loop_state exists and is active: ensure it is fully saved to state.json.
3. Stage any unstaged tracked files that are part of the current task.
4. If there are staged changes: commit with message "chore: concert-push handoff".
5. Push current branch to origin.
6. Output summary: branch, commits pushed, quality_loop_state if saved.
7. Output next steps for continuing in the other environment.

**User guidance content:**
- This is a CLI command (`npx @he3-org/concert push`), NOT an agent command
- It works without an active LLM session — critical for when plan usage is exhausted
- It never loses work — safe to run at any point
- The output tells the user exactly which agent to run and what model to select

**Operating principles content:**
- Pure file and git operations — no LLM calls
- Never destructive: no force push, no branch deletion, no file deletion
- Fails safely if nothing to push

**Boundaries content:**
- This is a reference document, not an executable agent
- The actual push logic is in src/commands/push.ts
- Does NOT require an LLM session to execute

### concert-documenter.md

**Frontmatter:**
```yaml
name: concert-documenter
description: Phase-level documentation updater — updates higher-level docs after each phase completes
tools: Read, Write, Edit, Bash, Glob, Grep
model: balanced
interactive_only: false
```

**Role section content:**
Documentation specialist that runs after all tasks in a phase complete. Reads the PHASE-SUMMARY and all commits from the phase, then updates higher-level documentation: README sections, API docs, architecture docs, guides. Has a holistic view of all changes across all tasks in the phase — unlike individual coders who only see one task at a time.

**Workflow integration content:**
Spawned by concert-continue after all tasks in a phase complete. Part of CONCERT-WORKFLOW-EXECUTION.md. Produces a single conventional commit with all doc updates for the phase.

**Execution flow content:**
1. Read the PHASE-SUMMARY file for the completed phase.
2. Read all commits from the phase (via git log).
3. Identify higher-level docs that need updating: README.md, API docs, architecture docs, guides, any doc files referenced by the phase's tasks.
4. Update each doc file with relevant changes from the phase.
5. Produce a single conventional commit: `docs: update documentation for phase NN — <phase-name>`.
6. Do NOT update inline docs (those are handled by the coder during the quality loop).

**User guidance content:**
- Focus on higher-level docs, not inline code comments
- One commit for all doc updates in the phase
- Only update docs that are affected by the phase's changes
- Use the PHASE-SUMMARY as the primary source of what changed

**Operating principles content:**
- Holistic view: consider all tasks in the phase together
- Don't duplicate content — update existing sections rather than adding new ones
- Keep documentation consistent with the codebase
- Single commit for all doc changes

**Boundaries content:**
- Does NOT modify code files (only documentation files)
- Does NOT update inline code comments or JSDoc (that's the coder's job)
- Does NOT modify state.json (the runner handles state)
- Does NOT modify task files or PHASE-SUMMARY files
