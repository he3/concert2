## Detailed Instructions

### CONCERT-WORKFLOW-REVIEW-CYCLE.md

This workflow defines the review/accept/restart cycle used at every planning stage.

**Content to include:**

1. **Overview:** The review cycle applies to all planning stages (vision, requirements, architecture, UX, tasks). Each stage produces a draft, which goes through review before being accepted.

2. **Three commands in the cycle:**

| Command | Agent | Interactive | Action |
|---------|-------|-------------|--------|
| concert-review | concert-reviewer | yes | Two-phase review of the draft document |
| concert-accept | (no agent) | no | Accept draft, copy to *-SPEC.md, finalize stage (does NOT advance pipeline) |
| concert-restart | (no agent) | no | Discard draft, re-run the consultant agent |

3. **Two-phase review flow:**
   - Phase 1: "Do you have any changes?" — apply user changes one at a time
   - Phase 2: Reviewer's own concerns — present one at a time by severity (CRITICAL > IMPORTANT > SUGGESTION)
   - Counter format: `[1/N]` on each concern
   - Commit after each applied change

4. **Accept behavior:**
   - Copy the draft document to project-level `*-SPEC.md` (e.g., VISION.md -> docs/concert/VISION-SPEC.md)
   - Update state.json: `pipeline.<stage> = "accepted"` (do NOT advance `stage` — that is done by `/concert:continue`)
   - Update WIP PR body
   - Commit all changes
   - Prompt user to run `/concert:continue` to advance to the next stage

5. **Restart behavior:**
   - Update state.json: `pipeline.<stage> = "pending"`
   - The next `concert-plan` will re-run the consultant agent from scratch
   - The previous draft remains in the mission folder (overwritten by the new run)
   - No code rollback — restart only affects planning documents

### CONCERT-WORKFLOW-OBSERVABILITY.md

This workflow defines telemetry and logging rules.

**Content to include:**

1. **Telemetry logging:** After each successful task completion, append a record to `state.json` -> `telemetry[]`. Record schema:
   - task_file: string
   - task_index: number (1-indexed)
   - phase: number
   - model_assigned: "haiku" | "sonnet" | "opus"
   - confidence: "high" | "medium" | "low"
   - review_result: "PASS" | "NTH" | "MIN" | "MAJ" | "CRIT" | "none"
   - revision_count: number
   - skills_loaded: string[]
   - files_changed: number
   - completed_at: string (ISO 8601)

2. **Failure logging:** On task failure, append to `state.json` -> `failure_log[]`. Record includes: phase, task_file, task_index, error_type, error_summary, failed_at.

3. **History logging:** After significant events (stage transitions, task completions, failures, phase completions), append to `state.json` -> `history[]`. Each entry: `{ "event": "string", "description": "string", "timestamp": "ISO 8601" }`.

4. **Confidence assessment rules:**
   - HIGH: tests cover all acceptance criteria, pattern is familiar, model tier is appropriate
   - MEDIUM: tests cover most criteria, some uncertainty in approach
   - LOW: limited test coverage, unfamiliar pattern, or model tier may be insufficient

5. **Cost estimation:** Rough per-stage estimates stored in `state.json` -> `cost.by_stage`. These are rough estimates, not token counts. Updated after each stage.

### CONCERT-WORKFLOW-SELF-IMPROVEMENT.md

This workflow defines the post-mission retrospective process.

**Content to include:**

1. **Trigger:** Runs after successful verification, if `self_improvement.enabled` is true in concert.jsonc.

2. **Agent:** concert-verifier performs this analysis as a secondary step (or a separate invocation could be added).

3. **Input data:**
   - Telemetry records from state.json
   - Failure log from state.json
   - COST-REPORT.md
   - PHASE-SUMMARY files
   - Current agent, skill, and workflow definitions

4. **Output:** `CONCERT-IMPROVEMENT.md` in the mission folder with:
   - Prioritized suggestions (HIGH/MEDIUM/LOW priority)
   - Each suggestion cites specific telemetry evidence
   - "What Worked Well" section: ONLY highlights outcomes that exceeded expectations or were unexpected pleasant surprises worth codifying — does NOT list things that simply worked as expected
   - Suggested changes to agent definitions, skills, or workflows

5. **Constraint:** The retrospective agent NEVER modifies Concert files directly. It only produces a suggestion document for human review.
