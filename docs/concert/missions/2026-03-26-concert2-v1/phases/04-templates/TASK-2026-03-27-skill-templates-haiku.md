## Objective

Create the default skill files that ship with Concert. Skills provide domain-specific guidance to agents — they declare file patterns they apply to and contain coding standards, conventions, and patterns that agents should follow when working with matching files. Concert ships with a small set of foundational skills; users add project-specific skills after initialization.

## Files

- `templates/docs/concert/skills/typescript-standards/SKILL.md`
- `templates/docs/concert/skills/cli-ux-guidelines/SKILL.md`
- `templates/docs/concert/skills/agent-authoring/SKILL.md`

## Requirements

## Skill File Format

Every skill file must start with the managed header:

```
<!-- AUTO-GENERATED BY CONCERT v0.1.0 — DO NOT EDIT. This file is managed by Concert and will be overwritten on `concert update`. -->
```

Then YAML frontmatter:

```yaml
---
name: <skill-name>
applies_to: ["<glob-pattern>", "<glob-pattern>"]
---
```

Then markdown content with the skill's guidelines.

## Detailed Instructions

### typescript-standards/SKILL.md

**Frontmatter:**
```yaml
name: typescript-standards
applies_to: ["**/*.ts", "**/*.tsx"]
```

**Content guidelines to include:**
- Use `node:` prefix for Node.js built-in imports (e.g., `import * as fs from "node:fs"`)
- Prefer `const` over `let`, never use `var`
- Use explicit return types on exported functions
- Use TypeScript interfaces over type aliases for object shapes
- Use `unknown` over `any` where possible
- Error handling: use structured error types, not raw strings
- Use `node:path` for all path operations (never string concatenation)
- Use `node:fs` async methods where appropriate, sync in CLI commands
- Imports: group by external, internal, types — separated by blank lines
- Naming: camelCase for functions/variables, PascalCase for types/interfaces/classes
- File naming: kebab-case for files (e.g., `copy-helpers.ts`)
- Export: prefer named exports over default exports
- Tests: use `describe`/`it` structure, test file naming: `*.test.ts`

### cli-ux-guidelines/SKILL.md

**Frontmatter:**
```yaml
name: cli-ux-guidelines
applies_to: ["src/commands/**/*.ts", "src/cli.ts"]
```

**Content guidelines to include:**
- All CLI output follows the structure: header, body, next steps
- Success output goes to stdout, errors/warnings go to stderr
- Exit codes: 0 (success), 1 (error), 2 (usage error)
- No color, no emoji, no Unicode special characters
- All output fits within 80 columns
- Next steps are always present and actionable
- Error format: "Error: what\n  why\n\n  Fix:\n    how"
- Warning format: "Warning: what\n  details\n\n  Options:\n    options"
- Use 2-space indentation for body content
- File paths are repo-relative (not absolute)
- Commands are shown inline, unquoted

### agent-authoring/SKILL.md

**Frontmatter:**
```yaml
name: agent-authoring
applies_to: ["docs/concert/agents/**/*.md", "templates/docs/concert/agents/**/*.md"]
```

**Content guidelines to include:**
- Every agent file starts with the managed header comment
- YAML frontmatter must include: name, description, tools, model, interactive_only
- Six XML sections required: `<role>`, `<workflow_integration>`, `<execution_flow>`, `<user_guidance>`, `<operating_principles>`, `<boundaries>`
- Role section: define the agent's expertise and perspective
- Workflow integration: how this agent fits into the pipeline
- Execution flow: step-by-step numbered instructions the agent follows
- User guidance: how the agent communicates with the user
- Operating principles: invariants the agent always follows
- Boundaries: what the agent explicitly does NOT do
- Every agent output must end with "Next steps:" section
- State.json changes must be committed after mutations
- Interactive agents must check interactive_mode before proceeding
- Tool list should be minimal — only tools the agent actually needs
