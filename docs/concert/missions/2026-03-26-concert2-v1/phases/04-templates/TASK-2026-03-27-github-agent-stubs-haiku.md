## Objective

Create the GitHub Agent stub files in `templates/.github/agents/`. Each stub is a minimal markdown file that GitHub Agents UI discovers and displays in the agent selector. The stub contains only the agent description and an instruction to read the full agent definition from `docs/concert/agents/`. This three-file pattern (full definition + GitHub stub + Claude Code command) ensures the full agent definition is the single source of truth.

## Detailed Instructions

Every GitHub Agent stub file follows this exact template. Copy it exactly for each agent, replacing only the `<name>` and `<description>` placeholders.

### Stub file template

```markdown
<!-- AUTO-GENERATED BY CONCERT v0.1.0 — DO NOT EDIT. This file is managed by Concert and will be overwritten on `concert update`. -->

---
description: "<description>"
---

Read the full agent definition and follow its instructions exactly:

Read docs/concert/agents/concert-<name>.md
```

### Agent names and descriptions

Use the exact descriptions below for each agent stub:

| Agent | Description |
|-------|-------------|
| concert-init | Interactive mission interviewer — creates vision and initializes mission state |
| concert-analyst | Requirements analyst — generates comprehensive requirements from accepted vision |
| concert-architect | Architecture planner — designs system architecture from accepted requirements |
| concert-designer | UX designer — creates UX specifications from accepted architecture |
| concert-planner | Task decomposer — breaks approved plans into executable phases, waves, and task files |
| concert-coder | TDD implementer — writes tests first, implements, verifies all tests pass |
| concert-code-reviewer | Code quality reviewer — reviews diffs against acceptance criteria with severity ratings |
| concert-continue | Execution orchestrator and session continuation — starts execution, resumes after crashes/timeouts, or advances to next stage |
| concert-reviewer | Two-phase reviewer — applies user changes first, then presents reviewer concerns one at a time |
| concert-status | Cold-start recovery — reads state.json and displays full mission status with next steps |
| concert-debug | Scientific-method debugger — forms hypotheses and tests them systematically to fix failures |
| concert-verifier | Acceptance tester — verifies completed work against requirements |
| concert-quick | Quick task executor — handles small tasks without the full pipeline |
| concert-push | Cross-environment handoff — documents the push command behavior for agents |
| concert-documenter | Phase-level documentation updater — updates higher-level docs after each phase completes |
