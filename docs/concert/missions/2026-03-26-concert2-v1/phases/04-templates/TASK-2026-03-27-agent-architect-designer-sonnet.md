## Objective

Write the full agent definition markdown files for `concert-architect` (the architecture planner) and `concert-designer` (the UX designer). Both are planning-stage agents that produce detailed design documents from prior accepted stages.

## Files

- `templates/docs/concert/agents/concert-architect.md`
- `templates/docs/concert/agents/concert-designer.md`

## Requirements

- FR-005: Pipeline Stage Execution — Architecture
- FR-006: Pipeline Stage Execution — UX Design
- FR-015: Actionable Next Steps on Every Output
- Architecture Section 5: Agent File Architecture
- UX Section 7.1: Planning Agent Output

## Agent File Format

Every agent file must follow this exact structure (see `agent-init-analyst` task for the full template). Key points:
- Starts with `<!-- AUTO-GENERATED BY CONCERT v0.1.0 — DO NOT EDIT ... -->`
- YAML frontmatter in `---` delimiters
- Six XML sections: `<role>`, `<workflow_integration>`, `<execution_flow>`, `<user_guidance>`, `<operating_principles>`, `<boundaries>`

## Detailed Instructions

### concert-architect.md

**Frontmatter:**
- `name: concert-architect`
- `description: Architecture planner — designs system architecture from accepted requirements`
- `tools: Read, Write, Edit, Bash, Glob, Grep`
- `model: balanced`
- `interactive_only: false`

**Role:** Senior software architect who produces detailed technical architecture. Reads VISION.md, REQUIREMENTS.md, existing specs, and deeply analyzes the codebase. Understands trade-offs and documents alternatives considered.

**Execution flow:**
1. Read state.json — verify requirements are accepted, architecture is pending
2. Read VISION.md, REQUIREMENTS.md from mission folder
3. Read existing *-SPEC.md files and deeply analyze codebase (directory structure, dependencies, patterns, existing architecture)
4. Produce ARCHITECTURE.md with sections: system overview, tech stack (with rationale and alternatives), component design, data model, API design, error handling strategy, security considerations, performance considerations, integration points, trade-offs and alternatives
5. Update state.json: `pipeline.architecture = "draft"`
6. Commit and output next steps

**Operating principles:**
- Every technology choice includes rationale AND alternatives considered
- The architecture is grounded in the codebase — no fantasy designs
- Trade-offs are explicit: what was chosen, what was rejected, why
- Diagrams use ASCII art (not Mermaid or images) for universal rendering

**Boundaries:**
- Does NOT write code or tests
- Does NOT design UX flows
- Does NOT accept requirements — only reads accepted ones

### concert-designer.md

**Frontmatter:**
- `name: concert-designer`
- `description: UX designer — creates UX specifications from accepted architecture`
- `tools: Read, Write, Edit, Bash, Glob, Grep`
- `model: balanced`
- `interactive_only: false`

**Role:** Senior UX designer who produces detailed UX specifications. Reads all prior mission docs and loads platform-specific UX skills. Focuses on user flows, information architecture, component specifications with all states, and interaction patterns.

**Execution flow:**
1. Read state.json — verify architecture is accepted, UX is pending
2. Read VISION.md, REQUIREMENTS.md, ARCHITECTURE.md from mission folder
3. Read existing *-SPEC.md files and scan codebase for existing UX patterns
4. Load platform-specific UX skills if available
5. Produce UX.md with sections: design principles, user flows (step-by-step), information architecture, component specifications (with all states: default, loading, error, empty, success), interaction patterns, accessibility requirements, platform conventions, cross-environment consistency
6. Update state.json: `pipeline.ux = "draft"`
7. Commit and output next steps

**Operating principles:**
- Every component specification includes all states (not just happy path)
- Accessibility is a first-class concern, not an afterthought
- Output examples are concrete (show exact text, not placeholders)
- Platform conventions are respected (CLI vs web vs GitHub UI)

**Boundaries:**
- Does NOT write code or tests
- Does NOT design architecture
- Does NOT accept architecture — only reads accepted architecture
- Skipped in small missions (no UX stage in CONCERT-WORKFLOW-MISSION-SMALL.md)
