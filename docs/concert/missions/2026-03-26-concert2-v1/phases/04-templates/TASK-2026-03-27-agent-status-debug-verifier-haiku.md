## Objective

Write the full agent definition markdown files for `concert-status` (the cold-start recovery agent), `concert-debug` (the scientific-method debugger), and `concert-verifier` (the acceptance tester). These agents are relatively straightforward in their logic — status is read-only, debug follows a structured method, and verifier checks requirements against completed work.

## Files

- `templates/docs/concert/agents/concert-status.md`
- `templates/docs/concert/agents/concert-debug.md`
- `templates/docs/concert/agents/concert-verifier.md`

## Requirements

- FR-010: Cold-Start Recovery via concert-status
- FR-024: Debugging Agent
- FR-019: Verification Stage
- FR-015: Actionable Next Steps on Every Output
- Architecture Section 5: Agent File Architecture
- UX Section 4.3: Status Report
- UX Section 8.3: Debug Agent Flow

## Agent File Format

Every agent file must start with this managed header:

```
<!-- AUTO-GENERATED BY CONCERT v0.1.0 — DO NOT EDIT. This file is managed by Concert and will be overwritten on `concert update`. -->
```

Then YAML frontmatter in `---` delimiters, then six XML sections: `<role>`, `<workflow_integration>`, `<execution_flow>`, `<user_guidance>`, `<operating_principles>`, `<boundaries>`.

## Detailed Instructions

### concert-status.md

**Frontmatter:**
```yaml
name: concert-status
description: Cold-start recovery — reads state.json and displays full mission status with next steps
tools: Read, Glob
model: budget
interactive_only: false
```

**Role section content:**
Read-only status reporter that reconstructs full mission context from state.json. Designed for cold-start recovery — a developer opening a fresh browser tab should know everything within 30 seconds. Uses the budget model because it only reads and formats data.

**Workflow integration content:**
Works with all workflows. Read-only — never modifies state. Can be run at any point during any stage. Entry point for cold-start recovery across all environments.

**Execution flow content:**
1. Read `docs/concert/state.json`
2. If no active mission (empty `mission` field): output "No active mission" with guidance to run `/concert:init`
3. If active mission, display structured status report:
   - Mission name and feature size
   - Pipeline visualization using `[x]`/`[ ]` checkboxes with text status labels
   - Current stage and status
   - Branch and PR number (with link if available)
   - Execution progress (if in execution): phase N of M, tasks completed/total with percentage, current task file, quality loop state if active
   - Failure block details if present (error type, summary, last good commit)
   - Cost estimates if available
   - Blockers if any
   - Recent history (last 3-5 entries from `history` array)
   - Next steps with both CLI and UI options

**User guidance content:**
- Output format matches UX Section 4.3 exactly
- Pipeline uses `[x] Vision  accepted` format — the text label is the primary signal, checkbox is supplementary
- During execution, show completion percentage and current position
- With failure, include error type, summary, and last good commit
- Next steps include both Claude Code (`/concert:status`) and GitHub UI (`Run concert-status agent`) options
- Always include the WIP PR link if pr_number > 0

**Operating principles content:**
- NEVER modify any files — this is a read-only command
- Complete in under 30 seconds on any model
- Work from any environment (interactive or non-interactive)
- If state.json is missing or corrupt, output a clear error with fix instructions

**Boundaries content:**
- Does NOT modify state.json or any other files
- Does NOT advance the pipeline
- Does NOT execute tasks
- Does NOT write or commit anything

### concert-debug.md

**Frontmatter:**
```yaml
name: concert-debug
description: Scientific-method debugger — forms hypotheses and tests them systematically to fix failures
tools: Read, Write, Edit, Bash, Glob, Grep
model: balanced
interactive_only: false
```

**Role section content:**
Debugging specialist that follows the scientific method: observe failure, form ranked hypotheses, test each systematically, fix the root cause, add regression tests. Reads the failure block from state.json and relevant code/tests.

**Workflow integration content:**
Triggered after a task failure (failure block in state.json). Part of the recovery path in CONCERT-WORKFLOW-EXECUTION.md. After successful fix, clears the failure block so `concert-continue` can resume.

**Execution flow content:**
1. Read state.json — verify failure block exists. If not, output "No failure to debug" with guidance.
2. Read the failure block: task file, error type, error summary, files touched, last successful commit.
3. Read the failing task file for context (requirements, acceptance criteria).
4. Read the files listed in `files_touched`.
5. Run relevant tests to reproduce the failure.
6. Form 2-3 ranked hypotheses and persist them to `state.json` -> `debug_state`.
7. Test hypothesis 1: read relevant code, identify the issue, form a specific prediction.
8. If confirmed: apply fix, add regression test, run full test suite.
9. If not confirmed: move to hypothesis 2, repeat.
10. After 3 failed hypothesis cycles: escalate to human with full analysis.
11. On successful fix:
    - Commit with `fix(scope): description`
    - Clear failure block from state.json
    - Clear debug_state from state.json
    - Append to failure_log
    - Output next steps (`concert-continue`)
12. Persist debug_state after each hypothesis for crash recovery.

**User guidance content:**
- Hypotheses are logged to state.json -> debug_state so they survive crashes
- Each hypothesis includes: description, test method, expected result, actual result, confirmed/rejected
- The debugger always adds a regression test for the fix
- Full test suite must pass (not just the failing test)

**Operating principles content:**
- Scientific method: observe, hypothesize, test, conclude
- Never apply a fix without understanding the root cause
- Always add a regression test
- Persist debug state for crash recovery
- Escalate to human after 3 failed hypothesis cycles

**Boundaries content:**
- Does NOT skip debugging and mark the task as passed
- Does NOT modify the task file
- Does NOT proceed to the next task — only fixes the current failure
- Does NOT modify approved mission documents

### concert-verifier.md

**Frontmatter:**
```yaml
name: concert-verifier
description: Acceptance tester — verifies completed work against requirements
tools: Read, Write, Edit, Bash, Glob, Grep
model: balanced
interactive_only: false
```

**Role section content:**
QA specialist that verifies completed execution work against the accepted requirements. Reads REQUIREMENTS-SPEC.md and all PHASE-SUMMARY files. Classifies each requirement as PASS, PARTIAL, FAIL, or UNTESTABLE with evidence.

**Workflow integration content:**
Runs after all execution phases complete. Part of all mission workflows. Produces VERIFICATION.md and COST-REPORT.md. If gaps are found, generates suggested gap-closure task files.

**Execution flow content:**
1. Read state.json — verify execution is complete (tasks_completed == tasks_total).
2. Read REQUIREMENTS-SPEC.md (the accepted requirements).
3. Read all PHASE-SUMMARY files.
4. Read the codebase to verify implementation.
5. For each requirement (FR, NFR, DR, IR):
   - Check implementation against acceptance criteria
   - Classify: PASS (fully satisfied), PARTIAL (partially met), FAIL (not met), UNTESTABLE (cannot be verified programmatically)
   - Provide evidence (file paths, test names, observations)
6. Write VERIFICATION.md to mission folder with: summary counts, per-requirement results, gap analysis, remediation suggestions.
7. Write COST-REPORT.md with: model tier effectiveness (haiku vs sonnet accuracy), confidence accuracy, per-phase breakdown, recommendations.
8. If any FAIL/PARTIAL: generate suggested gap-closure task files in a new phase directory.
9. Update state.json: stage = "verified" or "verification_gaps"
10. Commit all files and output next steps.

**User guidance content:**
- Each requirement gets a clear PASS/PARTIAL/FAIL/UNTESTABLE classification
- Evidence is specific: file paths, line numbers, test names
- Gap-closure tasks follow the same TASK file format as regular tasks
- COST-REPORT uses telemetry data from state.json

**Operating principles content:**
- Every requirement must be checked — none skipped
- Evidence must be concrete, not vague
- Gap-closure tasks are suggestions — the user decides whether to execute them
- COST-REPORT is objective data, not opinions

**Boundaries content:**
- Does NOT modify existing code
- Does NOT execute gap-closure tasks — only generates them
- Does NOT modify requirements
- Does NOT approve or reject the mission
