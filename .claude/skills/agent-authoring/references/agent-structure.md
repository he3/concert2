## Agent Definition Structure

### File Location

Agent definitions live at `.claude/agents/<agent-name>.md`. The filename must use lowercase letters, numbers, and hyphens only.

### Required YAML Frontmatter

```yaml
---
name: <agent-name>
description: <one-line purpose — what this agent does and when to use it>
---
```

Field reference:

- `name` — matches the filename (without `.md`). Lowercase, hyphens, 1-64 chars.
- `description` — concise purpose statement. Used for routing and display.

### Required Sections

Use XML-style tags to structure the agent body. Each section serves a specific purpose and **must appear in this exact order**:

1. **Managed header** (HTML comment) — `<!-- AUTO-GENERATED BY CONCERT ... -->`
2. **YAML frontmatter** — between `---` delimiters
3. **Environment check** (interactive agents only) — before `<role>`, same format as `.concert/templates/environment-check.md`
4. **`<role>`** — Who the agent is. 2-4 sentences covering identity, responsibilities, and working style. This sets the behavioral frame.
5. **`<operating_principles>`** — Table of constraints: `| # | Principle | Constraint |`. Use `ALWAYS` or `NEVER` — no `SOMETIMES`.
6. **`<boundaries>`** — Explicit list of what the agent does NOT do. Prevents scope creep. Use `NEVER` statements.
7. **`<workflow_integration>`** — Boot sequence: numbered list of files to load in the order specified by `.concert/templates/boot-sequence.md`. Always read `state.json` first.
8. **`<execution_flow>`** — Step-by-step numbered instructions. Each step must be unambiguous. Include explicit failure handling (see `references/tools-and-error-recovery.md`). Do NOT embed workflow decision logic here — reference workflow files instead.
9. **`<user_guidance>`** — Reference `.concert/templates/user-guidance.md` for output templates. Always include both `/concert:X` and `(@concert-X in Copilot)` in next-steps blocks.

Section ordering rationale: role, principles, and boundaries are stable content that rarely changes — placing them first maximizes prompt cache hits. Workflow integration and execution flow are more volatile and belong lower.

### GitHub Agent Stub

Every agent definition needs a companion stub at `.github/agents/<agent-name>.agent.md`. The stub is a lightweight redirect — it contains only metadata and a `Read` instruction pointing to the full definition.

**CRITICAL:** YAML frontmatter `---` MUST be on line 1. Copilot CLI requires strict frontmatter format for agent discovery. No HTML comments before frontmatter.

**Standard (non-interactive) format:**

```markdown
---
name: '<agent-name>'
description: '<same description as the full definition>'
---

<!-- ⚠️ AUTO-GENERATED BY CONCERT — DO NOT EDIT
     This file is managed by Concert and will be overwritten on `concert update`.
     Any manual changes will be lost. To customize behavior, see .concert/README.md -->

Read .claude/agents/<agent-name>.md for your complete instructions.
Read .concert/state.json for current position.
<one-line context about what the agent should do>
```

**Interactive format** (for agents that require user input):

```markdown
---
name: '<agent-name>'
description: '<description> (requires interactive mode)'
---

## Environment Check (MUST RUN FIRST)

Check if you can interactively prompt the user for input.
If you CANNOT ask the user questions (e.g., GitHub Agents UI):

STOP IMMEDIATELY and output:
❌ This command requires interactive mode (Claude Code CLI, CoPilot CLI, or web).
Run `/concert:<command>` in an interactive CLI instead.

Do NOT attempt to proceed without user input. Do NOT guess at answers.

## If Interactive

Read .claude/agents/<agent-name>.md for your complete instructions.
```

Critical rules for stubs:

- YAML frontmatter `---` MUST be on line 1 — no HTML comments before it
- Do NOT include `tools:` or `model:` in stubs — stubs are metadata-only
- The `Read` instruction must point to the correct `.claude/agents/` path
- Description should match the full definition (may be slightly shorter)

### Alias Agents

When a slash command name differs from the canonical agent name (e.g., `/concert:review` maps to `concert-reviewer`), create an alias agent:

```markdown
<!-- AUTO-GENERATED BY CONCERT — DO NOT EDIT. ... -->

---

name: concert-review
description: Command alias — delegates to concert-reviewer

---

This is a command alias that maps `/concert:review` to the concert-reviewer agent.

Read `.claude/agents/concert-reviewer.md` for your complete instructions and follow them exactly.
```

The alias exists because Claude Code maps `/concert:X` to `.claude/agents/concert-X.md` by convention. The GitHub stub (`concert-review.agent.md`) can directly reference the canonical agent.

### Context Optimization

Agents are part of the LLM prompt — shorter agents leave more room for task content. Follow these guidelines:

- Keep agent definitions under 120 lines (budget agents under 50 lines)
- Place stable content (role, principles, boundaries) at the top for cache hits
- Do NOT duplicate workflow logic in agents — reference workflow files instead
- Do NOT duplicate next-steps templates — reference `user-guidance.md` instead
- Do NOT embed severity definitions, commit formats, or other shared content inline
- Follow the boot sequence order from `.concert/templates/boot-sequence.md` for cache-friendly file loading

### Token Budget

- Keep agent definitions under 200 lines. If it's longer, the agent has too many responsibilities — split it.
- Place stable content (role, principles, boundaries) at the top. Place volatile content (execution flow with conditional branches) lower.
- Do not repeat information available in referenced files (skills, specs). Use `Read` instructions instead.

### Rules Must Stay Light

Rules (`.claude/rules/`) load in every context. Keep them minimal:

- Only add rules for truly universal conventions (e.g., commit format)
- Never add agent-specific logic to rules
- Prefer skills or workflow files for domain-specific knowledge
